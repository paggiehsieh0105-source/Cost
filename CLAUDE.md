## 二、核心計算邏輯（必須在Python中完整重現）

### 2.1 整體公式
```
單件總成本 = 原物料成本小計 + 直接人工成本小計 + 製造費用 + 外包加工費小計 + 物流運費小計

未稅報價 =
    IF 毛利率模式 = "成本基礎(加成法)":   單件總成本 × (1 + 毛利率)
    IF 毛利率模式 = "售價基礎(毛利率法)"： 單件總成本 ÷ (1 − 毛利率)

最終報價 =
    IF 報價含稅 = "含稅":  未稅報價 × (1 + 營業稅率)
    ELSE:                 未稅報價

預估月營收 = 最終報價 × 預估月產量
```

⚠️ **毛利率計算模式可隨時切換，兩種公式都必須實作，不可只做一種。**

---

### 2.2 原物料成本（含包材）

```
原料總成本 = 原料單位用量 × 原料單價 × (1 + 原料損耗率)     # 簡化為1筆（整組配方視為一筆）

包材小計(每項) = 包材用量 × 包材單價 × (1 + 包材損耗率)
原物料成本小計 = 原料總成本 + Σ(包材小計)
```

**包材單價來源：** 不是手動輸入，是從「包材主檔」依「包材名稱」查詢取得（見第四節）。

---

### 2.3 直接人工成本

人工成本採用「批量回推法」：先用整批生產的總時間回推每件加工時間，再計算工資。

```
每件加工時間(分) = 批量總時間(分) ÷ 批量總件數

時薪 = 作業員月薪 ÷ (每月工作天數 × 每日工作時數)        # 預設 22天 × 8小時
       — 此時薪為全廠共用值，可在「每一個製程」上以手動覆寫的方式個別調整

人工小計(每製程) = 每件加工時間 × (時薪 ÷ 60) × 參與人數
直接人工成本小計 = Σ(人工小計)
```

可能涵蓋的製程：配方攪拌、充填、貼標、裝箱等，筆數不固定（須支援動態新增/刪除）。

---

### 2.4 製造費用（按比例攤提，非明細攤提）

```
製造費用 = (原物料成本小計 + 直接人工成本小計) × 製造費用攤提率
```

⚠️ 這是**簡化設計**：不追蹤機台折舊/水電/房租明細，直接用一個可調整的百分比抓出大概金額。

---

### 2.5 外包加工費

```
外包加工費小計 = Σ(外包單價 × 數量)   — 多筆，動態列表
```

---

### 2.6 物流運費（多列加總，可混用不同運送方式）

運送方式共 3 大類：**專車 / 物流 / 棧板**

```
每件運費 =
    IF 運送方式 = "專車":  查「專車區域對照表」(依縣市分區) 取得單價
    ELSE:                  查「物流／棧板對照表」取得單價

該列小計 = 出貨數量 × 每件運費
物流運費小計 = Σ(該列小計)     — 多列，一次出貨可分批用不同運送方式，全部加總
```

## 四、包材主檔（未來資料庫的核心資料表，極重要）

這是**獨立的主檔資料**，不是某個專案的附屬資料，因為同樣的包材（例如同一款瓶身）會被用在多個不同代工專案上。

### 資料庫表設計（建議）
```sql
CREATE TABLE packaging_materials (
    material_id   TEXT PRIMARY KEY,   -- 例如 PKG-001
    material_name TEXT NOT NULL,      -- 例如 瓶身-30ml噴瓶
    spec          TEXT,               -- 規格描述
    unit          TEXT,               -- 個/張/箱
    unit_price    REAL NOT NULL,
    vendor        TEXT,
    note          TEXT,
    updated_at    TEXT
);
```

## 七、開發中的重要決策記錄（避免之後重複討論）

1. **毛利率計算模式必須兩種都支援**，使用者可隨時切換，不是寫死一種。
2. **製造費用採用比例攤提制**（佔原料+人工成本的%），不做機台折舊/水電明細管理。
3. **原料簡化為1筆**（整組配方視為一個總價），包材才逐項列出。
4. **時薪由月薪反推**（月薪÷22天÷8小時），不直接輸入時薪；但允許個別製程手動覆寫。
5. **包材單價不可手動輸入**，必須從包材主檔查詢取得，避免同一包材在不同地方填錯價格。
6. **物流運費支援多列、可混用三種運送方式**（專車/物流/棧板），因為一次出貨可能分批。
7. 專車依5個縣市分區報價，**目前單價皆為0，需業主提供實際報價後才能正確試算**。
8. **【重要修正】物流運費必須先「分攤到每件」才能併入單件成本**：物流運費每列的小計（出貨數量×每件運費）是**整批/整單的運費總額**，不是單件成本。正確算法：
   ```
   物流運費分攤後(單件) = Σ(出貨數量×每件運費) ÷ 實際出貨總數量
   單件總成本 = 原料+人工+製造費用+外包 + 物流運費分攤後(單件)
   ```
   **分攤分母的兩種情境（`CostProject.shipping_order_quantity`）：**
   - **情境A：拆批出貨**（部分用專車、部分用物流），各列「出貨數量」加總本身就等於總出貨量 → 不需特別設定，系統會自動把各列數量加總當分母
   - **情境B：多種方式合併計費同一批貨**（例如：專車跑一趟＋棧板另計件，兩者都是針對同一批1000件貨）→ 此時各列數量單位不一致（趟數 vs 件數），加總當分母會算錯，**必須明確設定 `shipping_order_quantity = 1000`**，系統才會用這個值當分母，而非加總各列數量
   範例（情境B）：專車2,000元 + 棧板280元 ＝ 總額2,280元，這批貨實際1,000件 → 分攤後每件 **2.28元**（不是用2行加總的數量去除）。
   ⚠️ **Excel雛形尚未套用此修正**，目前只有Python版本（`cost_model.py`／Streamlit App）是正確的。
9. **【2026-06-23修改】原料簡化**：`RawMaterial` 移除損耗率欄位，公式簡化為 **整組配方成本(unit_price) × 件數(quantity)**，不再有損耗率乘項。資料庫的 `raw_material_loss_rate` 欄位仍保留(向下相容)但不再使用。
10. **【2026-06-23新增】紙箱用量（依尺寸計費）**：新增 `CartonLine(size, quantity)`，尺寸費率存在 `Settings.carton_rates`（資料庫表 `carton_sizes`），費用會**併入物流運費總額**（`shipping_total = 各運送方式小計 + 紙箱小計`），並用同一套分攤邏輯（`shipping_order_quantity` 或「各運送方式數量＋紙箱數量」加總）換算成每件分攤金額。預設尺寸：小箱(20x20x20cm)15元、中箱(30x30x30cm)25元、大箱(40x40x40cm)35元，皆可在「共用設定」分頁調整。

    **修正方式**：改成先用「名稱」查詢資料庫裡有沒有同名的既有資料——有就直接更新那一筆（**編號維持原本的，不會被新編號取代**），沒有才依照 `material_id` 新增/更新。這樣無論使用者填不填編號，都不會同時撞到兩個UNIQUE限制。

20. **【2026-06-25修正：極重要的Python/pandas陷阱】NaN在 `or` 判斷式裡是「真值」**：
    `st.data_editor` 的空白格子在pandas裡會變成 `NaN`，但 `bool(float('nan'))` 在Python裡是 `True`（因為NaN不等於0，也不等於任何值，包括它自己）。所以像這樣的寫法：
    ```python
    float(row.get("單價") or 0)   # ❌ 錯誤：欄位空白時這裡是 NaN，不是 0
    ```
    在欄位空白時，`NaN or 0` 的結果還是 `NaN`（不會變成0），因為 `or` 只在左邊是「假值」時才取右邊，而NaN是真值。接著把NaN存進SQLite的 `NOT NULL` 欄位時，sqlite3的Python驅動會把NaN當成NULL，丟出 `sqlite3.IntegrityError: NOT NULL constraint failed`。
    **修正方式**：全面改用 `_safe_float()` / `_safe_int()` / `_safe_str()` 三個輔助函式（定義在 `app.py` 開頭），明確用 `pd.isna()` 檢查後才給預設值，不再用容易出錯的 `or` 寫法。所有從 `data_editor` 讀資料轉成 dataclass 的地方（`df_to_packaging`、`df_to_labor`、`df_to_subcontract`、`df_to_shipping`、`df_to_carton`、包材主檔儲存、紙箱費率儲存）都已經改用這套安全函式。
    ⚠️ **這個坑很隱密**，因為對「字串」欄位用 `or ""` 通常看起來沒問題（NaN字串化後肉眼可能不會馬上發現），但對「數字」欄位（尤其是存進NOT NULL欄位時）就會直接炸掉。**未來新增任何讀取 `data_editor` 數字欄位的程式碼，都必須用 `_safe_float`/`_safe_int`，不能用 `row.get(x) or 預設值` 這種寫法。**

## 十三、2026-06-25 重大計算邏輯調整

這次調整牽動 `cost_model.py`／`database.py`／`app.py` 三個檔案，重點如下：

### 1. 新增「損耗率」設定（共用設定頁）
`Settings` 新增三個欄位：`raw_material_loss_rate`、`labor_loss_rate`、`shipping_loss_rate`（都是小數，0.05=5%）。
套用方式：**算完各自的成本小計之後，乘以(1+損耗率)當整體緩衝**：
```
原物料成本 = (原料+包材小計) × (1+原物料損耗率)
人工成本   = 人工小計 × (1+人工損耗率)
物流運費(分攤後) = (運費總額÷訂單量) × (1+物流損耗率)
```
這個損耗率跟包材明細裡「每一項包材自己的損耗率」是**不同層級**：包材的損耗率是個別物料的損耗，這個新的損耗率是「算完一整個類別小計之後」再多抓一筆緩衝。

### 2. 原料(RawMaterial)簡化：移除「件數」欄位
原本 `RawMaterial(quantity, unit_price)`，現在只剩 `RawMaterial(unit_price)`，公式變成：
```
原料成本小計 = 整組配方成本（不再乘件數）
```
使用者直接輸入「每件成品的整組配方成本」，這個數字本身就是單件原料成本，不需要再乘任何數量；「訂單量」只會在最後「成本加總與報價試算結果」的「預估月營收」計算時才用到（`monthly_revenue = price_final × monthly_quantity`，這個邏輯本來就有，沒有變動）。

### 3. 移除「外包加工費」整個模組
`SubcontractLine` 類別、`CostProject.subcontract_lines`、`CostBreakdown.subcontract_total` 全部刪除，不再使用。原因：使用者實際填外包明細的「數量」代表整張訂單的總數量，導致跟物流運費同樣的「整批金額誤當單件成本」風險，與其再修一次分攤公式，業主決定直接不用這個模組。
資料庫的 `project_subcontract` 表已從 schema 移除（新建的資料庫不會有這張表；舊資料庫如果已經建過這張表，會留著沒人讀寫，不影響運作）。

### 4. 物流／紙箱費用分攤分母統一簡化為「訂單量」
移除了 `CostProject.shipping_order_quantity` 這個獨立欄位，以及「合併計費同一批貨」的checkbox／複雜分母判斷邏輯。現在不管是物流運費還是紙箱費用，**分攤分母永遠是 `project.monthly_quantity`（預估月產量／訂單量）**：
```
物流運費(分攤後) = (各運送方式小計 + 紙箱小計) ÷ 訂單量 × (1+物流損耗率)
```
表格裡填的「出貨數量」「用量(箱)」只用來算「這一列自己的費用是多少」，跟分攤的分母（訂單量）是兩件獨立的事，不會互相影響，邏輯更單純好懂。
資料庫的 `projects.shipping_order_quantity`、`projects.raw_material_qty` 欄位仍留在schema裡（舊資料庫相容用），但新版程式不會再讀寫這兩欄。

## 十四、2026-06-25修正：刪除的「紙箱尺寸」「專車區域」會被自動補回來

**問題**：使用者在「共用設定」頁刪除預設的「小箱／中箱／大箱」尺寸並儲存後，重新整理頁面，發現又被加回來了；同樣的問題也會發生在「專車區域」。

**根本原因**：`database.py` 的 `init_db()` 在 `app.py` 開頭**每次頁面重新整理都會被呼叫一次**（不是只有第一次建資料庫時才呼叫）。裡面用 `INSERT OR IGNORE` 把預設的紙箱尺寸／專車區域寫進資料庫——`OR IGNORE` 只會擋掉「重複的主鍵」，並不知道「這筆資料是使用者刻意刪除的」，所以每次重整頁面，已經被刪除的預設項目又會被重新插入一次。

**修正方式**：`init_db()` 改成先查詢 `shipping_regions`／`carton_sizes` 這兩張表**目前是不是完全空的**，只有在「完全空」（代表這是全新資料庫，第一次啟動）時才寫入預設值；只要表裡已經有任何資料（不管是預設值還是使用者自己加的），就不會再嘗試寫入預設值。這樣使用者刪除任何一筆，重新整理頁面也不會被復原。
已寫測試重現此問題並驗證修正（模擬：建資料庫→刪除「小箱」→模擬重新整理頁面再呼叫一次`init_db()`→確認小箱不會被加回來）。

## 十五、2026-06-25再次修正：包材明細改回「雙表格」設計（找到Streamlit的硬限制）

第24項曾經把包材明細的「編號／名稱／用量／損耗率」合併成單一表格（用disabled欄位+每次重算名稱），實測發現有個**Streamlit本身的硬限制**：

> **disabled（不可編輯）欄位的「即時重新計算」，只對「本來就存在的列」有效；使用者在這次編輯過程中「新增的列」，它的disabled欄位值一旦建立就固定下來，之後不管程式重新傳值、重新計算，畫面都不會更新，永遠停留在剛新增時的空白狀態。**

實測現象：讀取既有專案時，原本就存在的包材列名稱可以正確即時顯示；但使用者在編輯過程中新增的列，名稱永遠顯示「None」，無法修正。這不是程式邏輯錯誤，是Streamlit `data_editor` 元件對「新增列」與「既有列」的內部狀態管理機制不同所致。

**最終修正方式（改回雙表格設計）**：
- **上方：可編輯表格**，只有「包材編號」「用量」「損耗率」三欄（不含名稱），「用量」「損耗率」都設定 `default` 值（1.0／0.0），新增列不會整排顯示刺眼的「None」
- **下方：唯讀的即時預覽表格**（`st.dataframe`），逐列把「包材編號」對應的「包材名稱」秀出來——因為這是**獨立的、每次重新執行都從頭算的`st.dataframe`**，不依賴`data_editor`的「新增列/既有列」內部狀態追蹤機制，所以**不論新列舊列都能正確即時顯示**，沒有上述的硬限制問題
- `df_to_packaging()` 不受影響，仍然只讀「包材編號」「用量」「損耗率」三欄

⚠️ **重要教訓**：之後如果還想嘗試把「自動計算的disabled欄位」放進同一個可編輯的`data_editor`裡，要先確認所有列都是「既有列」（例如讀取已存檔的專案），如果使用者會在編輯過程中動態新增列，這個技巧就會在新增列上失效，應該改用「另一個獨立的唯讀表格」顯示計算結果，而不是塞進同一個可編輯表格裡。

## 十六、2026-06-25新增：包材明細「用量」欄位加上明確說明，避免填錯成訂單總數量

**問題**：使用者實際使用時，把包材明細的「用量」誤填成「這次訂單總共要用多少個包材」（例如填1000代表訂單1000件），而不是「一件成品實際使用量」（通常是1）。這會讓原物料成本被灌爆成天文數字，因為公式是 `用量 × 單價 × (1+損耗率)`，這裡的「用量」設計上就是「單件用量」。

**修正方式**：
- `app.py`：「用量」欄位的顯示標籤改成「用量(一件成品的實際使用量)」，並在欄位上方加上明顯的警語：「⚠️ 用量請填一件成品的實際使用量，不是訂單總數量——訂單總共要採購多少，系統會自動用『用量×預估月產量』幫你算」
- `cost_model.py`：`PackagingLine.quantity` 的docstring也補上同樣的說明，方便之後維護的人一看就懂，不會又改錯

這個問題跟之前修正過的「物流運費」「外包加工費」是同一類陷阱的變形：**使用者直覺上想填「這批/這次訂單的數量」，但系統設計上某些欄位的「數量」其實是「單件」的概念**。如果之後還有類似欄位（例如人工製程的「參與人數」、紙箱的「用量(箱)」），都要留意說明文字是否清楚區分「單件」vs「整批/整單」，避免同樣的混淆再次發生。

## 十七、2026-06-25最終調整：包材明細改回直接用「名稱」選擇，不再讓使用者面對編號

**背景**：第18~24項陸續把包材明細從「名稱選擇」改成「編號選擇＋自動帶出名稱」，目的是解決編號/名稱的唯一性問題，但過程中發現Streamlit的硬限制（新增列的disabled欄位無法即時更新），體驗一直不夠好。使用者最終決定：**直接恢復用「包材名稱」選擇，不要再出現編號**，並且選好名稱後要自動帶出單價/規格供確認。

**最終設計**：
- `PackagingCatalog` 新增 `all_names()`（取得所有名稱，重複的只列一次）、`find_by_name(name)`（依名稱找第一筆符合的資料）、`find_all_by_name(name)`（依名稱找出所有符合的資料，用於名稱重複時的提示）
- 「成本試算」表單的包材明細：可編輯表格欄位變成「包材名稱」（`SelectboxColumn`，可搜尋）、「用量」、「損耗率」，**不再有「包材編號」這一欄**
- 下方維持原本的唯讀預覽表格設計（沿用第15項的雙表格架構，避免Streamlit新增列的限制），但內容改成顯示「包材名稱、編號、規格、單價」——如果同名稱對應到多筆資料，會在「編號」欄位提示「⚠️名稱重複共N筆，會採用編號XXX這一筆」
- `df_to_packaging()` 改成依「包材名稱」用 `catalog.find_by_name()` 查詢，找不到就跳出警告並忽略該列；找到後，底層 `PackagingLine` 仍然儲存 `material_id`（不是名稱），維持資料庫關聯設計不變
- **包材主檔頁（維護包材清單的地方）維持不變**，仍然有「包材編號」欄位，因為那裡是在管理主檔資料本身，編號還是唯一識別碼；只有「成本試算」表單裡，使用者選擇要用哪個包材時，介面上不會再看到編號

**設計取捨**：如果同一個名稱在主檔裡對應到多筆不同編號的資料（例如不同供應商的同款瓶蓋），選擇名稱時系統會**自動採用第一筆找到的資料**，並在預覽表格提示重複筆數與實際採用的編號，讓使用者可以發現異常並自行到「包材主檔」調整名稱讓它們不要重複，或接受這個簡化的行為。

## 十八、2026-06-25三項連鎖修正：舊資料庫殘留欄位導致存檔失敗、設定刪除失效

使用者實際使用時回報三個問題，調查後發現第2、3項其實是同一個根源造成的連鎖反應：

### 1. `project_packaging.material_name` 仍是 NOT NULL（造成存檔錯誤）
雖然第21項已經把 schema 改成用 `material_id`，但**先前的migration只是「多加一個material_id欄位」，沒有真正解除舊的`material_name NOT NULL`限制**（SQLite的`ALTER TABLE ADD COLUMN`沒辦法移除既有欄位的限制）。新版程式存檔時已經不會填`material_name`，所以撞到這條殘留的舊規則，丟出`sqlite3.IntegrityError: NOT NULL constraint failed: project_packaging.material_name`。
**修正**：在 `_migrate_missing_columns()` 裡新增一段「整張表重建」邏輯——只要偵測到 `project_packaging` 還有 `material_name` 欄位，就建一張全新的表（只有真正需要的欄位）、把舊資料搬過去（用既有的`material_id`，捨棄`material_name`）、刪除舊表、改回原本的表名。已寫測試模擬真實的舊資料庫狀態（material_id和material_name共存、material_name仍NOT NULL），驗證重建後存檔不再出錯，且舊資料正確保留。

### 2. 讀取既有專案編輯，資料會遺漏（其實是第1項問題的連鎖效應，不是獨立的bug）
`save_project()` 是包在一個資料庫交易（transaction）裡：人工、外包(已移除)、物流、紙箱、包材明細依序寫入，最後才一次性`commit()`。**如果中途任何一筆寫入失敗（例如撞到第1項的NOT NULL錯誤），因為從來沒有呼叫到`commit()`，整筆交易裡「已經寫但還沒確認」的所有修改都會被捨棄**，不會真的存進資料庫。所以使用者看到的「資料遺漏」，其實是那次儲存根本沒有成功，畫面顯示的還是上一次成功儲存的舊版本。**修好第1項之後，這個問題會一併消失，不需要額外修正。**

### 3. 「共用設定」頁刪除的紙箱尺寸／專車區域會自動跑回來（這次才找到真正原因）
之前（第14項）以為是 `init_db()` 重複補預設值造成的，雖然那確實也是個問題並已修正，但**真正主要的原因其實是 `save_settings()` 的寫入邏輯只有 `INSERT...ON CONFLICT`（新增/更新），從來沒有刪除過任何資料**。使用者在表格裡刪除一列、按下儲存，資料庫裡那一列根本沒有被刪掉，只是沒有被覆蓋而已。
**修正**：改成「先清空、再整批寫入」（`DELETE FROM shipping_regions` / `DELETE FROM carton_sizes`，再依目前畫面上的內容整批 `INSERT`），確保資料庫內容跟畫面上的表格完全同步，使用者刪除的項目才會真正消失。已寫測試驗證：刪除「小箱」存檔後，重新讀取確認小箱不在清單裡。

### 包材明細表格左側的數字欄位是什麼？
這是 Streamlit `st.data_editor` **內建的列號／選取把手**，不是使用者輸入的資料欄位，純粹用來方便勾選整列、刪除整列、或拖曳調整順序，不影響任何計算邏輯，不需要額外處理。
